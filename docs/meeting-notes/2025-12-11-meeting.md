## 회의 안건
1. AWS 개발 일정 및 비용 관리

## 회의 내용

### 안건 1: AWS 개발 일정 및 비용 관리
- **논의 내용**:
  - 현재 AWS 인프라(예: Terraform로 올려둔 환경)를 사용 중이며, 클라우드 자원을 쓰기 시작한 만큼 비용도 함께 모니터링해야 한다는 의견 제기
  - 전체 개발 일정은 실제 기능 개발은 12월 29~30일 전후까지 마무리하는 것을 목표로 하고, 이후 남은 기간(말일 이후)은 발표 자료 작성, 화면 캡처 및 정리, 간단한 수정 및 테스트에 집중하는 일정
  - 무분별한 AWS 사용으로 비용이 과다 발생하지 않도록, 예상 사용 기간·규모에 맞는 비용 관리가 필요


### 안건 2: WBS 기반 프로젝트 관리 및 AI 파트 계획
- **논의 내용**: 
  - 프로젝트 전체 일정은 WBS를 기준으로 관리하고 있으며, AI 관련 작업은 실제 개발을 진행하면서 세부 태스크를 하나씩 추가·보완하는 방식으로 계획 중
  - 현재 잡혀 있는 WBS 대비 어디까지 진행되었는지 진행률을 한 번 점검해 볼 필요가 있음
  - Terraform은 AWS 리전, 사용할 클러스터 버전 등 EKS 관련 설정 정보가 담겨 있는 코드이고 디버깅 및 설정 관리용 IaC로 이해하면 됨
  - 일정이 생각보다 여유롭지 않으므로, 남은 기간 동안의 작업 우선순위를 WBS 기준으로 재정비해야 함
  - 생성에 필요한 요소들을 코드에 정의해 두었고 Terraform 설치 후 명령 실행만으로 AWS에 동일한 EKS 환경을 재생성할 수 있는 구조
  - 특히 AI 파트는 “일단 구현하면서 WBS에 추가”하는 방식이라 중간에 누락되거나 일정이 밀릴 위험이 있어, 현재까지의 작업을 WBS에 재정렬하는 작업이 필요

### 안건 3: 개발 진행 상태(사전 작업·개발 이슈)
- **논의 내용**:
  - 사전 준비/구성 관련 작업(기획·전처리·환경 세팅 등)과 주요 개발 파트 모두 현재까지는 큰 이슈 없이 진행 중
  - 개발적인 부분은 팀을 신뢰하고 맡기고 있으며, 일정 준수 여부가 가장 중요함
  - 기술적 난관보다는 “정해진 기간 안에 WBS 상 태스크를 소화할 수 있는지”가 핵심 리스크

### 안건 4: 패키지 구조 및 구현 시간 관리
- **논의 내용**:
  - 백엔드/AI 쪽의 패키지 구조를 제대로 보려면, 실제 코드와 기능이 어느 정도 쌓여야 함
  - 남은 4주(또는 스프린트 4 기준) 동안은 구현 속도를 최대한 끌어올려, 기능을 빠르게 채워 넣는 것이 패키지 구조 정리에도 도움이 될 것
구현 패턴을 쉽게 찾을 수 있을 것
  - 구조 설계와 리팩터링도 중요하지만, 현재 시점에서는 우선 “돌아가는 기능을 확보하는 것”이 우선

- **결정 사항**: 
  - AWS 비용 모니터링 체계 마련: 현재 사용 중인 AWS 자원(클러스터, 스토리지 등)에 대해 월 추정 비용을 확인하고, 비용 알림 또는 주간 체크 루틴을 정함 (Infra) (2025-12-10~2025-12-17)
  - 개발·발표 일정 마일스톤 확정: 기능 개발 완료 s목표(12월 말), 발표 자료 작성·테스트 기간을 포함한 상세 마일스톤을 정리(PM / 전체) (2025-12-10~2025-12-14)
  - WBS vs 실제 진행률 점검: 기존 WBS와 실제 작업 진행 상황을 비교해, 누락된 작업 또는 일정 지연 구간을 체크하고 우선순위를 조정 (PM / 각 파트 리더) (2025-12-11~2025-12-16)
  - AI 파트 세부 WBS 보완: AI 관련 작업(모델 선택·성능 테스트·파이프라인 구현 등)을 세부 태스크로 쪼개어 WBS에 명시적으로 반영 (AI) (2025-12-11~2025-12-18)
  - 패키지 구조·구현 전략 정리: 남은 기간 동안 구현 속도를 높이기 위한 단위 기능 쪼개기, 패키지 구조 가이드(간단한 규칙)를 정리 (BE / AI) (2025-12-12~2025-12-20)
