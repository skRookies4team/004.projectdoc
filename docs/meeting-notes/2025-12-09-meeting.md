## 회의 내용

### 안건 1: UI 코드 처리 방향
- **논의 내용**:
  - 현재 UI는 기본적인 형태까지 구현 완료된 상태이며, 추후 시간이 남으면 계속 개선·수정
  - 이번 스프린트/발표의 핵심은 기능·연동·AI/인프라이므로 UI 코드는 “필요 최소한만 동작하게 해두고 대대적인 리팩터링은 이후 과제로 넘기자”는 방향으로 논의 중

### 안건 2: AWS EKS 클러스터 설정 및 Terraform 코드 구성
- **논의 내용**: 
  - EKS 클러스터 설정은 이미 완료된 상태
  - 클러스터를 띄우면 로드 밸런서, 노드 그룹 등 관련 리소스가 자동 생성되도록 설정되어 있으며, 노드 생성·프로비저닝에 시간이 다소 소요되는 점을 확인함
  - Terraform은 AWS 리전, 사용할 클러스터 버전 등 EKS 관련 설정 정보가 담겨 있는 코드이고 디버깅 및 설정 관리용 IaC로 이해하면 됨
  - 생성에 필요한 요소들을 코드에 정의해 두었고 Terraform 설치 후 명령 실행만으로 AWS에 동일한 EKS 환경을 재생성할 수 있는 구조
  - 콘솔로 수동 설정하는 대신 코드 기반(IaC)으로 환경을 재현할 수 있다는 점이 장점으로 강조됨

### 안건 3: IAM 계정·Access Key·대시보드 접근 이슈
- **논의 내용**:
  - EKS 및 AWS 리소스 관리를 위해 IAM 계정과 Access Key가 필요하다는 점이 다시 확인됨
  - 현재는 IAM 대시보드(관리 콘솔)에 접근이 잘 되지 않는 상태로, 권한(permission) 문제로 인해 접근이 막혀 있는 상황으로 보임
  - 기본 VPC/서브넷 구성은 기본 서브넷 4개가 자동 생성되도록 설정됨
  - 대시보드 접근 권한 문제가 해소되지 않으면 키 발급, 권한 관리, 모니터링 등에 제약이 크므로 IAM/대시보드 접근 권한 정리 작업이 필요함

### 안건 4: 토큰 관리 및 프론트단 처리 방식
- **논의 내용**:
  - 다른 조들은 AI·LLM 호출 시 토큰 관리를 적극적으로 활용하고 있으며, 우리 팀도 인증/인가 토큰을 어떻게 다룰지 정리할 필요
  - 기본 구조: 프론트에서 백엔드로 요청을 보낼 때, HTTP 헤더(예: Authorization)에 토큰을 담아 전달하는 방식 사용
  - React 기준 프론트단에서는 토큰·세션 정보를 어디에 두고 관리할지에 따라 구조가 달라질 수 있음
    선택지로는 React Context, Redux(리덕스) 두 가지가 주요 옵션으로 언급됨
  - 토큰 처리·저장 방식은 “React 토큰 관리, context vs redux” 등을 참고하면 구현 패턴을 쉽게 찾을 수 있을 것
  - 시연·데모에 필요한 최소 인증 구조를 빠르게 만들고, 이후 확장성·보안성을 고려해 구조를 다듬는 전략이 필요

- **결정 사항**: 
  - UI 코드 정리 범위 확정: 이번 발표까지는 필수 UI만 유지하고 UI 코드 대대적 리팩터링은 이후 과제로 넘기되, 현재 구조에서 꼭 수정이 필요한 부분만 최소 범위로 수정 (FE) (2025-12-10~2025-12-16)
  - EKS·Terraform 설정 검증: 기존 Terraform 코드 기반으로 EKS 클러스터 재생성 테스트를 진행하고, 로드 밸런서·노드 그룹 등 자동 생성 리소스 구성이 정상인지 확인 (Infra) (2025-12-09~2025-12-15)
  - IAM 계정·대시보드 접근 권한 정리: IAM 계정 및 Access Key 발급 상태, 대시보드 접근 불가 이슈를 정리하고 필요한 권한을 요청·설정해 콘솔 접근을 가능하게 함 (Infra) (2025-12-09~2025-12-18)
  - 토큰 관리 방식 확정 및 적용: React 기준으로 토큰 저장·전달 방식(Context vs Redux 등)을 결정하고, Authorization 헤더로 백엔드에 전달하는 구조를 구현·테스트 (FE / BE) (2025-12-11~2025-12-20)
