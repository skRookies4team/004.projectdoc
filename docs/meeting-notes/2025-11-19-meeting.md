## 회의 안건
1. 챗봇이 교육 영상 재생을 직접 수행하는 구조 검토
2. 챗봇 클릭 기반 페이지 이동과 챗봇 직접 재생 방식의 차이 정리
3. 4대 법정 교육(연 1회) vs 직무 교육(연 수회) 업데이트 주기 관련 멘토 피드백 대응
4. 초기 구현 범위(챗봇 / 전처리 테스트) 결정

## 회의 내용

### 안건 1: 질문 그룹핑 및 FAQ 역할 재논의
- **논의 내용**:
  - 챗봇 기반 영상 재생 구조
  - “교육 틀어줘”라고 사용자가 말하면 챗봇이 자동으로 해당 교육 영상을 찾아 재생하는 방식 논의
  - 기존 방식은 사용자가 플로팅 아이콘 클릭 → 교육 페이지 이동 후 재생하는 구조였음
  - 새로운 방향은 챗봇이 메인 허브 역할을 하여 재생 요청 시 바로 콘텐츠를 찾아 보여주는 방식
  - 챗봇이 영상 재생 명령·라우팅까지 수행하려면 콘텐츠 메타데이터 관리 필요

### 안건 2: 챗봇 클릭과 페이지 이동 구조
- **논의 내용**: 
  - 강사님 질문: “클릭해도 다 챗봇 형태로 되나요?”
  - 플로팅 아이콘 클릭 → 교육 메뉴 진입점만 챗봇
  - 그 이후 실제 화면은 동일한 교육 페이지
  - 단, “교육 틀어줘”라고 챗봇에게 말하면 챗봇이 저장된 정보를 기반으로 해당 영상을 바로 틀어줌
  - 챗봇 인터페이스와 교육 페이지 UI의 역할 분리 필요
  - UX 혼잡 방지를 위해 ‘챗봇이 어디까지 개입할 것인지’ 범위 명확히 해야 함

### 안건 3: 멘토 의견에 대한 우리팀의 내부 정리
- **논의 내용**:
  - 멘토 질문: “4대 법정교육은 1년에 1번인데, 왜 굳이 주요 교육 축으로 넣었는가?” “직무 교육은 1년에 여러 번 바뀌는데, 바뀔 때마다 다시 학습시키는가?”
  - 멘토 질문은 “교육 변경 주기가 빠른데 그걸 시스템 구조에서 어떻게 처리할 것인가?”라는 본질적 우려
  - 이를 해결하는 방향으로 문서 자동 전처리/임베딩 파이프라인 필요성 대두
  - 따라서, 단기적으로는 챗봇 기능과 문서 전처리(RAG 전처리) 정확도 확보에 집중하기로 함

### 안건 4: 초기 구현 범위 확정
- **논의 내용**:
  - 교육 전체 시스템을 한 번에 구현하는 대신, 우선 챗봇 기능 + 자료 전처리 테스트를 1차 목표로 삼기기로 결정
  - 교육 콘텐츠 자동 재생, 직무 교육 자동 업데이트는 2차 스프린트 이후로 이동
  - 시간 내 완성도 확보 위해 핵심 기능부터 좁혀서 개발 필요

- **결정 사항**: 
  - 챗봇 영상 재생: 챗봇 명령 기반 영상 재생 로직 MVP 구현 (FE/AI) (2025-11-20)
  - UI 구조: 챗봇 진입점 + 교육 페이지 분리 설계 (FE) (2025-11-20)
  - 전처리 테스트: RAG 전처리 파이프라인 PoC(정확도 테스트) 우선 개발 (AI) (2025-11-19)
  - 교육 콘텐츠 업데이트: 직무 교육 자동 업데이트는 스프린트 2로 이관 (PM/기획) (2025-11-20)
  - 스코프 확정: 스프린트 1 범위: 챗봇 중심 + 교육 재생 요청 처리로 확정 (PM) (2025-11-19)
