## 회의 내용

### 안건 1: 개발 환경 현황 및 설계 일정
- **논의 내용**:
  - 현재 팀원별 로컬 개발 환경은 모두 세팅 완료된 상태
  - 프론트엔드는 우선 퍼블리싱 작업 위주로 진행 중이며, 백엔드 코드는 아직 로컬 환경에서만 동작하는 수준
  - 개발 절차·설계 관련 산출물(아키텍처, 설계 데이터 등)은 목요일 내로 정리·작성할 계획
  - 로컬 환경 세팅 이후, 설계 산출물(WBS, 아키텍처, API 설계 등)이 실제 개발 일정과 맞물리도록 이번 주 안에 정리 필요

### 안건 2: MSA로 가는 이유 및 서비스 축약
- **논의 내용**: 
  - 서비스 구조는 MSA 패턴으로 가져가는 방향으로 정리
  - “MSA로 간다”는 것에 대한 이견이 없었으며, 실제 운영 수준의 풀 MSA까지는 아니더라도 형식·패턴은 MSA에 맞춰 마무리할 가능성이 높음
  - ‘유저(직원) 쪽 서비스’는 관리자 영역에서의 사용자/계정 관리 기능과
역할 분리가 생길 것 같아 이번 서비스 축약 대상에서는 우선순위를 낮춰 제외하는 방향으로 논의
  - 어떤 기능을 기준으로 4개 서비스로 나눌지, MSA 기준(도메인, 책임, 데이터 경계)을 문서화할 필요 있음

### 안건 3: 전처리·OCR·RagFlow 활용 방향
- **논의 내용**:
  - 전처리 파이프라인은 내부 파서(parser)를 직접 커스터마이징하여 서비스에서 요구하는 포맷으로 데이터를 적재·연계할 수 있도록 구성하는 방향으로 작업을 진행 중임
  - 외부에서 생성한 ‘OCR 결과(슬라이드형 PDF 등)’를 별도 데이터셋으로 만들고, 이를 RagFlow 안에 넣어 RagFlow의 knowledge base 형태로 동작시키는 방안을 시도 중
  - 정리하면, 전처리·OCR은 우리 쪽에서 커스터마이징, 결과 데이터셋은 RagFlow에 적재, RagFlow는 벡터DB/KB로 돌아가게 하는 구조로 설계
  - 전처리·OCR 결과를 RagFlow에 넣을 때, 데이터 스키마·메타데이터 구조를 어떻게 맞출지가 핵심 포인트

### 안건 4: AI·프론트엔드 데모 방향 및 리소스 고려
- **논의 내용**:
  - 서비스 분류와 게이트웨이를 통해 요청이 자동으로 분배되고, 알고리즘이 완벽하지 않더라도 “자동화되어 돌아가는 모습”을 보여주는 것이 1차 목표
  - 이미지 생성 등 LLM·비전 모델을 사용할 때 메모리 부족 이슈가 발생할 수 있으므로 리소스 사용량을 고려한 기능 범위 설정이 필요
  - 프론트엔드는 “이게 AI 기반 서비스이고, 4조의 최종 결과물이다”가
2분 정도의 데모 영상/시연으로 잘 전달되면 된다는 식으로, 최소 기능 + 전달력 중심으로 방향 수립
  - 프론트는 모든 기능을 완벽히 다 넣기보다, 핵심 플로우가 끊기지 않고 자연스럽게 이어지는 시연 시나리오에 집중할 필요

### 안건 5: 프론트 퍼블리싱 이슈·브랜치 관리 방식
- **논의 내용**:
  - 현재 브랜치를 자주 생성하며 작업 중이라는 의견
  - 프론트 퍼블리싱 관련 이슈를 만들 때, front-layout, 3주차-4주차, sprint-2-2, #95 등과 같이 스프린트/주차/레이아웃 기준으로 이슈 이름을 구조화해서 관리 중
  - Git 브랜치와 이슈를 1:1 혹은 논리적으로 연결되게 관리하면 추후 변경 이력 추적에 도움이 됨
  - 프론트 관련 이슈·브랜치 네이밍 규칙을 팀 공통 규칙으로 문서화해두면 좋음

### 안건 6: 프로젝트 개요 작성 방향
- **논의 내용**:
  - 프로젝트는 2주 단위(스프린트 단위)로 진행·정리하는 방식이 바람직함
  - 프로젝트 개요 문서 중요 항목 세 가지: 프로젝트 선정 배경, 활용 방안, 기대 효과
  - 이후 산출물(요구사항, 설계, 개발, 테스트 등)은 이 개요를 토대로 2주 단위로 끊어서 정리하는 구조를 추천
  - 프로젝트 한 번에 컨셉을 전달하는 문서이므로 “무엇을 왜, 어떤 효과를 기대하고 하는지”에 초점을 맞춰 작성해야 함

- **결정 사항**: 
  - MSA 서비스 분할 정리: 핵심 기능을 기준으로 4개 서비스로 축약하고, 각 서비스의 역할·경계·데이터 책임을 간단한 다이어그램과 텍스트로 정리 (BE / Infra / 기획) (2025-12-02~2025-12-06)
  - 전처리·OCR → RagFlow 연계 설계: 내부 파서·외부 OCR 결과를 어떤 스키마로 데이터셋화해 RagFlow에 넣을지, 샘플 데이터 기준으로 연계 플로우를 정의 (전처리 / AI) (2025-12-02~2025-12-09)
  - 프론트 이슈·브랜치 네이밍 규칙 문서화: 프론트 퍼블리싱·레이아웃 작업 시 사용할 이슈명·브랜치명 규칙을 Notion 또는 Git 문서로 정리 (FE) (2025-12-02~2025-12-05)
  - 프로젝트 개요 문서 초안 작성: 프로젝트 선정 배경, 활용 방안, 기대 효과를 포함한 프로젝트 개요 문서 초안을 작성 (PM / 기획) (2025-12-02~2025-12-08)
