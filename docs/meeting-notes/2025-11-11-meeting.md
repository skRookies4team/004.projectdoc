## 회의 안건
1. 기능명세서와 비기능명세서의 구분 및 작성 기준
2. 그룹웨어 대체 방안(Outlook, ERP 연동 등) 검토
3. 요구사항-기능사항 매핑 구조 및 WBS 연계

## 회의 내용

### 안건 1: 기능명세서와 비기능명세서 구분
- **논의 내용**:
  - 기능명세서: 사용자가 요구한 기능(서비스, 기능 흐름 등)을 기술한 문서
    - 예시: “사용자가 로그인하면 어시스턴트 메인 화면이 열린다.”
  - 비기능명세서: 성능, 속도, 안정성, 보안성 등 품질 속성을 정의한 문서
    - 예시: “로그인 응답 시간은 2초 이내여야 한다.”
  - 비기능 요구사항은 성능 테스트 항목과도 연결될 수 있음

### 안건 2: 그룹웨어를 Outlook으로 대체하는 방안
- **논의 내용**: 
  - 자체 그룹웨어를 새로 개발할 경우, 개발 시간과 리소스 부담이 큼
  - Outlook은 Microsoft 365 기반으로, 일정 관리(Calendar), 메일/공지, Teams 연동, 파일 공유(OneDrive 연계) 등의 기업 그룹웨어 핵심 기능을 이미 내장
  - 그룹웨어 개발을 생략하고 이미 그룹웨어 기능을 모두 갖춘 Outlook을 활용
  - 기술적 연결:
    - Outlook 계정 기반으로 로그인 인증(Keycloak 연동 가능)
    - Outlook 메일·캘린더·Teams API를 통해 사용자 일정/이력 불러오기
    - 어시스턴트는 Outlook 내부 플러그인처럼 동작 (Floating Assistant)  

### 안건 3: ERP (e-count) 그룹웨어 연동 방안
- **논의 내용**:
  - ecount ERP에 회원가입하여 ‘그룹웨어·ERP 기능(UI 포함)’을 활용하고,
이를 어시스턴트의 백그라운드 인터페이스로 시각화 연동하는 방안 제시
  - 장점: 그룹웨어 개발 시간 단축, 실제 업무 시스템과 유사한 환경 구현 가능

### 안건 4: 요구사항과 기능사항의 관계 및 관리 방식
- **논의 내용**:
  - 요구사항(Requirement): 고객이 요청한 기능 및 서비스 니즈
    - 예시: “직원 교육 이수 현황을 자동으로 확인하고 보고”
  - 기능사항(Functional Spec): 요구사항을 실제 개발로 구현한 구체적 항목
    - 예시: “교육 데이터 API 연동 및 대시보드 시각화 기능 구현”
  - 두 항목을 요구사항-기능사항 매핑표로 관리해야 함
  - 기능요구사항은 ‘WBS(Task 단위)’로 세분화되어 스프린트별로 배분됨

- **결정 사항**: 
  - 기능/비기능 명세: 각 문서 구분 기준 확정 및 작성 템플릿 초안 제작 (기획 파트) (2025-11-11)
  - 그룹웨어 대체: Outlook UI 활용 여부 검토 및 저작권 확인 (프론트엔드 파트) (2025-11-12)
  - ERP 연동: ecount ERP 가입 후 UI 연동 테스트 진행 (백엔드/프론트엔드 파트) (2025-11-12)
  - 요구사항 매핑: 요구사항-기능사항 매핑 (기획/PM 파트) (2025-11-11)
