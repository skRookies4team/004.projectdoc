## 회의 안건
1. 교육 이수 현황 UI 및 그룹웨어 백그라운드 화면 구성
2. 교육 콘텐츠 플레이 방식(동영상 플레이어, 분할 영상, 예외 검증) 결정
3. 시청 여부 확인 기술(비전, 랜드마크 추출 등) 적용 가능성
4. 사용자(인사담당자 등)의 교육 콘텐츠 생성 기능 제공 여부
5. 그룹웨어 오픈소스 확보 가능성 및 대체 UI 방향
6. 차별화 전략: 교육 이수 기반 서비스 확장, 놀라움 요소(UX)
7. WBS 구성(스프린트 1/2/3 우선 정의)

## 회의 내용

### 안건 1: 교육 화면(백그라운드)의 방향성
- **논의 내용**:
  - “나의 교육 이수 현황”을 그룹웨어 백그라운드 화면처럼 띄우는 방식 논의
  - 교육 이수 현황 기반으로 차별화 전략 가능(예: 개인/부서 교육 리포트)
  - 행사 일정 등은 하드코딩으로 고정 페이지 구성 가능행사 일정 등은 하드코딩으로 고정 페이지 구성 가능
  - 페이지를 직접 구현할지, AI 자동 생성 페이지를 붙일지는 선택 필요
  - 그룹웨어처럼 보이게 할 UI가 필요하지만, 실제 기능까지 구현할 필요는 없음

### 안건 2: 교육 콘텐츠 플레이 방식
- **논의 내용**: 
  - 콘텐츠는 동영상 한 편이 아니라 목차 기반(Section 기반) 시청 구조 고려
  - 모든 목차를 봐야 “이수 완료”로 인식하는 구조
  - 플레이어를 직접 개발할지, 슬라이드/iframe 기반 영상 플레이만 할지 결정 필요
  - 시청률 조작 방지를 위한 기술 필요 (seek bar 조정 금지)
  - 영상 분할 방식(마이에듀처럼)은 구현 난이도가 높을 수 있음

### 안건 3: 시청 여부 검증(비전 요소)
- **논의 내용**:
  - 브라우저 카메라를 통한 랜드마크(얼굴·시선) 추출로 실제로 보고 있는지 판단하는 기술 검토
  - JavaScript 기반 가벼운 Vision 기능으로 구현 가능
  - 시간 여유가 있을 때 넣을 **추가 기능(Advanced Feature)**로 분류
  - 법적 개인정보 이슈 검토 필요

### 안건 4: 퀴즈를 통한 검증의 한계
- **논의 내용**:
  - 단순 퀴즈 기반 검증은 “영상 안 보고 푸는 사람” 존재 → 이수 요건으로 부적절함
  - 퀴즈는 보조 검증 수단으로만 활용
  - 시청률 + 퀴즈 조합이 기본이지만, 영상 시청 “증명” 기능이 핵심

### 안건 5: 콘텐츠 자동생성 UI(사용자 생성 교육 콘텐츠)
- **논의 내용**:
  - 인사담당자·교육 담당자가 직접 교육 콘텐츠를 생성할 수 있는 기능 논의
  - 파일 업로드 → 문단 단위 추출 → 문단별 영상 생성 → 캐릭터 모델 적용 → 콘텐츠 완성
  - “코딩 없이 교육 콘텐츠 제작”이라는 강력한 차별화 포인트
  - 구현 범위의 난이도가 높음 → MVP에서는 최소 기능만 적용할 가능성

### 안건 6: 그룹웨어 오픈소스 확보 여부
- **논의 내용**:
  - 그룹웨어 오픈소스 확보 어려움(백엔드와 강하게 결합된 구조가 대부분)
  - 따라서 UI만 참고하여 가짜 그룹웨어(백그라운드용) 자체 제작 가능성 높음
  - 진짜 그룹웨어 구현은 필요 없으며, 기업 인트라넷 느낌만 구현해도 충분

### 안건 7: UX 차별화 / 놀라움 요소
- **논의 내용**:
  - “질문하면 답변한다”는 것은 기본 → 차별화 요소 필요
  - 교육 이수 기반 개인화 알림, 리포트 자동 생성, AI 추천 등이 포함될 수 있음
  - 사용자가 “아이콘을 클릭하고 싶어지게 만드는” 요소 필요
  - 단순 기능보다 경험(Experience)이 설계의 핵심 포인트

 ### 안건 8: RAG 기반 응답 가능 여부
- **논의 내용**:
  - 콘텐츠 기반 문의에 대해 RAG 기반 답변 제공 가능
  - 교육자료·문서 추출하여 임베딩 후 답변하는 구조
  - 실제 데모에서 RAG 정확도가 보이도록 데이터를 어느 수준까지 구축할지 결정 필요

 ### 안건 9: WBS 구성 방향
- **논의 내용**:
  - WBS는 스프린트 1 → 스프린트 2 → 스프린트 3 우선 정의 후 각 단계별 세부기능 배치 방식으로 구성해야 함
  - 세부 업무보다 스프린트 구조를 먼저 잡는 것이 뼈대를 만들기 용이

- **결정 사항**: 
  - 시청 여부 검증: 랜드마크 추출 기능은 여유 시 추가(후순위) (AI)
  - 그룹웨어 UI: 오픈소스 미사용, 자체 백그라운드 UI 제작 (FE) (2025-11-14)
  - WBS: 1/2/3 스프린트 우선 정의 및 구조 확정 (PM) (2025-11-14)
